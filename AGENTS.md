# Правила работы с Ven4Tools

## Область

Правила относятся к WPF-клиенту, лаунчеру, каталогу, установщику и релизному процессу Ven4Tools.
Общие правила экосистемы находятся в `..\AGENTS.md`.

## Актуальная база

- Клиент: .NET 8, WPF, `win-x64`, self-contained.
- Стабильная версия на момент создания правил: `3.4.13`.
- Лаунчер: `2.0.0`.
- Основная ветка: `main`.
- Репозиторий: `https://github.com/Ven4ru/Ven4Tools`.
- Дополнительная внутренняя память проекта хранится вне репозитория.

Перед работой обязательно сверять эти данные с GitHub и файлами проекта.

## До правок

1. Выполнить:
   - `git status --short --branch`;
   - `git log -1 --oneline`;
   - проверку соответствия локальной ветки upstream.
2. Прочитать связанные с задачей файлы и тесты.
3. Для массовых или рискованных изменений создать снимок в:
   `C:\Users\Vench\_backups\Ven4Tools_<дата>_<время>\`.
4. Если задача затрагивает архитектуру, безопасность или инфраструктуру, явно описать риски и критерии проверки.
5. Субагентов использовать только по прямому запросу пользователя.

## Правила реализации

1. Сохранять текущую архитектуру и принятые паттерны, если задача не требует её изменения.
2. Не выполнять широкий рефакторинг вместе с функциональной правкой.
3. Не проглатывать исключения без логирования, если ошибка важна для диагностики.
4. Не блокировать UI-поток длительными операциями.
5. Для `HttpClient` использовать переиспользуемые экземпляры.
6. Для winget:
   - использовать `--disable-interactivity`;
   - не добавлять принудительный `--locale en-US`;
   - валидировать идентификаторы пакетов и аргументы.
7. Не восстанавливать удалённые ZapretService/ZapretModels.
8. Remote Install не возвращать в основной клиент до отдельного решения после реального пользовательского тестирования.
9. Механизм активации не должен автоматически выполнять сторонние скрипты; пользователь сохраняет контроль над действием.
10. Изменения пользовательского поведения сопровождать русским UI-текстом и обновлением локализации, если она затронута.

## Безопасность и приватность

1. Не коммитить реальные токены, пароли, private keys, cookies и персональные данные.
2. `Secrets.cs` должен оставаться локальным и игнорируемым.
3. Не добавлять credentials в PowerShell/Python-скрипты.
4. URL загрузок должны проходить проверку доверенного источника.
5. Загружаемые файлы проверять по SHA-256 там, где протокол его предоставляет.
6. Crash report и логи не должны содержать имя пользователя Windows, токены и сетевые пути с персональными данными.
7. Перед коммитом:

```powershell
C:\Users\Vench\Tools\gitleaks\gitleaks.exe dir . --no-banner --redact
git diff --check
```

## Запрещённые файлы

Не коммитить:

- `Secrets.cs`;
- `_release/`, `_publish*/`, `_backups/`;
- `bin/`, `obj/`, `.vs/`;
- `_deploy_env.ps1`;
- временные дампы и файлы `(2)`;
- credentials и диагностические скрипты с ними.

Если solution ссылается на локальный игнорируемый проект, это следует явно фиксировать в отчёте. Нельзя молча считать чистый clone собираемым.

## Сборка и проверка

Минимальная обязательная проверка:

```powershell
dotnet build .\Ven4Tools.sln -c Release --nologo
```

Критерий: 0 ошибок и 0 предупреждений.

Дополнительно по затронутой области:

- запуск соответствующих unit/UI-тестов;
- проверка JSON-каталога;
- тестирование offline/cache fallback;
- проверка авторизации и приватности;
- проверка лаунчера и механизма обновления;
- ручная проверка WPF-вёрстки при изменении XAML.

Не считать задачу завершённой только потому, что код компилируется.

## Git

1. Все сообщения коммитов писать по-русски.
2. Не добавлять в публичные материалы сведения об инструментах генерации и внутреннем процессе подготовки.
3. Коммитить только законченный проверенный блок.
4. Не использовать `--no-verify` для обхода проверок.
5. Не переписывать опубликованную историю.
6. Перед push проверить:
   - `git status`;
   - полный diff;
   - отсутствие секретов;
   - сборку 0/0;
   - корректность версии.

## Релиз клиента

1. Убедиться, что версия согласована в проекте, changelog и release notes.
2. Выполнить self-contained publish для `win-x64`.
3. Создать `Ven4Tools-Client-X.Y.Z.zip`.
4. Вычислить и записать SHA-256.
5. Клиентский релиз создаётся с тегом `vX.Y.Z` и может быть помечен latest.
6. После публикации проверить:
   - GitHub `main`;
   - тег релиза;
   - имя и хеш ассета;
   - CDN `version.json`;
   - фактическую загрузку и проверку хеша.
7. Тег, `main` и исходный коммит релиза должны указывать на одно состояние.

## Лаунчер

1. Текущая стабильная линия — `2.0.0`.
2. Тег лаунчера: `launcher-vX.Y.Z`; не помечать его latest вместо клиента.
3. Для исправлений 2.0.0 без новой версии заменять ассеты только по отдельному решению и проверять новый SHA-256.
4. Проверять оба ассета:
   - `Ven4Tools.Setup-X.Y.Z.exe`;
   - `Ven4Tools.Launcher-X.Y.Z.exe`.
5. Установка остаётся в `%LOCALAPPDATA%\Ven4Tools\Launcher\` без обязательного UAC.

## Сайт, API и CDN

1. Изменения сайта выполнять в отдельном репозитории `Ven4Tools-site`.
2. Изменения административной утилиты — в `ven4admin`.
3. Не редактировать production как замену коммиту в Git.
4. Не ломать контракт `api/db.php`, авторизацию и cookie/session-механику.
5. Помнить, что `.htaccess` не является гарантией безопасности на nginx-хостинге.
6. Перед деплоем проверять cache-busting CSS/JS.
7. Production-деплой, миграции БД и ротация секретов требуют отдельного подтверждения.

## Аудит и качество решений

1. При общем запросе на аудит уточнить его тип: security, code quality, dependencies, performance, UX, data, release, repository/Git или полный.
2. Для архитектурной инициативы оценить:
   - техническую корректность;
   - безопасность и приватность;
   - эксплуатацию, обновление и откат.
3. При явном запросе на мультиагентное ревью разделить эти направления между независимыми агентами и затем консолидировать результат.
4. Не предлагать низкоценную косметику, пока остаются подтверждённые проблемы стабильности, безопасности или релизного процесса.

## Завершение

Отчёт должен содержать:

- изменённые файлы и поведение;
- путь резервной копии;
- выполненные проверки и их результат;
- Git-статус и коммит, если он создан;
- что не проверено;
- оставшиеся риски и ручные действия.
