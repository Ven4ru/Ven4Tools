# Windows Update — управление обновлениями ОС в клиенте Ven4Tools

## Контекст

Сейчас клиент проверяет только обновления winget-приложений (`SystemTab.BtnCheckUpdates_Click`, вызывает `winget upgrade`). Обновлений самой Windows (кумулятивные/security-патчи, драйверы, необязательные/preview-обновления, feature-апдейты) клиент не касается вообще — это полностью новая подсистема.

Клиент запускается с манифестом `requestedExecutionLevel="requireAdministrator"` (`Ven4Tools/app.manifest`) — весь процесс всегда работает с правами администратора. Это ключевое архитектурное упрощение: никакой отдельной элевации/IPC для операций Windows Update не требуется, всё выполняется прямо в основном процессе.

## Цель

Полный цикл управления патчами Windows внутри клиента:
- поиск доступных обновлений всех типов (критичные, security, драйверы, необязательные/preview, feature);
- отображение категоризированным деревом с чекбоксами (чекбокс на категории выбирает все патчи внутри неё, tri-state при частичном выборе);
- скачивание и установка выбранных патчей;
- фоновая периодическая проверка с выбором режима (только уведомлять / уведомлять + скачивать в фоне) на первом входе в вкладку.

## Архитектура

### Компоненты

| Компонент | Назначение |
|---|---|
| `Ven4Tools/Views/Tabs/WindowsUpdateTab.xaml(.cs)` | Новая вкладка клиента, рядом с `CatalogTab`/`InstalledTab`/`SystemTab` и т.д. |
| `Ven4Tools/Services/WindowsUpdateService.cs` | Обёртка над `Microsoft.Update.Session` (`IUpdateSearcher`/`IUpdateDownloader`/`IUpdateInstaller`, COM interop). Поиск, скачивание, установка — всё в основном процессе (уже elevated). |
| `Ven4Tools/Services/WindowsUpdateBackgroundService.cs` | Периодическая фоновая проверка, по аналогии с `UpdateBackgroundService`. Режим поведения зависит от пользовательского выбора. |
| Новое поле в `AppSettings`/`ProfileService` | `WindowsUpdateMode: NotSet \| NotifyOnly \| NotifyAndDownload` — выбор пользователя на первом входе, персистентный. |

Прогресс скачивания/установки передаётся через `IProgress<T>` в UI-поток — тот же паттерн, что уже используется в `InstallationService` для прогресс-баров установки приложений из каталога. Отдельного JSON-файла статуса или межпроцессного взаимодействия не требуется, так как нет границы процессов (весь путь — поиск → выбор → скачивание → установка — идёт в одном доверенном elevated-процессе).

### Почему не нужен отдельный elevated-хелпер

Первая версия дизайна предполагала отдельный elevated-процесс (по аналогии с тем, как `OfficeTab`/`InstalledTab` через `Verb="runas"` элевируют конкретные внешние `.exe`) и JSON-файл статуса для IPC. От этого отказались после того, как подтвердилось, что сам клиент **всегда** запускается с правами администратора (`app.manifest`) — элевировать нечего, весь код и так исполняется с нужными правами. Это устраняет и IPC-сложность, и связанные с ней риски (ревалидация списка патчей между процессами, гонки на файле статуса).

### Технический подход к Windows Update API

Используется нативный **Windows Update Agent COM API** (`Microsoft.Update.Session`) — тот же API, которым пользуется сама Windows Update и WSUS-клиенты. Рассмотренные и отклонённые альтернативы:

- **PSWindowsUpdate** (PowerShell-модуль) — обёртка над тем же COM API, не даёт ничего дополнительного, но добавляет supply-chain зависимость (модуль не встроен в Windows, нужно тянуть с PowerShell Gallery или вендорить) — тот же класс риска, что уже принят как есть для Chocolatey `install.ps1` в security-аудите 2026-07-08. Отклонено — нет обоснования добавлять риск без выгоды.
- **usoclient.exe/wuauclt.exe** — устаревшие/недокументированные интерфейсы, не дают доступа к категориям и отдельным патчам. Не закрывает требование категоризированного выбора вообще. Отклонено.

## UI/UX-поток

**Первый вход на вкладку:** если `WindowsUpdateMode == NotSet` — показывается диалог выбора режима («Только уведомлять» / «Уведомлять и скачивать в фоне»), выбор сохраняется, больше не переспрашивается (сменить можно будет позже, вероятно из SystemTab).

**Основной layout:**
- Кнопка «Проверить обновления» + метка времени последней проверки.
- Дерево категорий (`IUpdate.Categories` — Security Updates, Critical Updates, Drivers, Feature Packs и т.д.), чекбокс на категории с tri-state, разворачивается в список патчей (чекбокс, KB-номер, заголовок, размер, важность).
- Строка «Выбрано: N патчей, X МБ».
- Кнопка «Установить выбранные» (неактивна, если ничего не выбрано или система занята — см. ниже).

**Установка:** клик → диалог подтверждения (список, общий размер, предупреждение о возможной перезагрузке, текст EULA для патчей с непринятой лицензией) → прогресс (текущий патч, фаза скачивание/установка, общий %) → сводка по завершении (успешно/не удалось, причина по каждому failed).

**Перезагрузка:** если хоть один установленный патч требует перезапуска — диалог «Требуется перезагрузка: Перезагрузить сейчас / Позже». Без таймера отложенного рестарта, без какой-либо автоматики — решение полностью за пользователем (явное требование).

**Фоновая проверка:** периодический таймер (как `UpdateBackgroundService`). Находит новые патчи → бейдж-счётчик на вкладке («Windows Update (N)») + запись в лог. В режиме `NotifyAndDownload` — дополнительно тихо скачивает в фоне (без установки, чтобы later-install шла быстрее — сразу фаза Install). Никогда не устанавливает автоматически без явного клика пользователя.

## Взаимная блокировка с установкой приложений

Подсистема Windows Installer (MSI) общая между установкой приложений из каталога и установкой части патчей Windows. Параллельный `msiexec` из обоих потоков может конфликтовать (тот же класс проблемы, из-за которого уже существует `InstallationService.InstallSemaphore` — ошибка Windows Installer 1618 при параллельных вызовах).

Решение — **симметричная явная блокировка**, а не тихое ожидание семафора:
- Если идёт установка/удаление приложения из каталога — кнопка «Установить выбранные» на вкладке Windows Update неактивна, с сообщением «Дождитесь завершения установки приложений».
- Если идёт установка патчей Windows — кнопки установки/удаления в каталоге неактивны, с сообщением «Дождитесь завершения установки обновлений Windows».
- Оба потока проверяют общий флаг «идёт системная установка» (на основе `InstallSemaphore.CurrentCount`) перед стартом, с понятным сообщением вместо непрозрачного зависания.

## Обработка ошибок и краевые случаи

- **Служба Windows Update отключена/не запущена** (`wuauserv`) — определяется при первом обращении к API, показывается сообщение с кнопкой «Запустить службу» (клиент elevated — может сам включить, но только по явному клику).
- **EULA у отдельных патчей** — если у выбранного патча есть непринятый EULA, текст показывается в диалоге подтверждения перед установкой; выбор чекбокса сам по себе не считается согласием с лицензией.
- **Reboot-pending от предыдущей партии** — Windows Update часто отказывается ставить новые патчи, пока не выполнен предыдущий перезапуск. Проверяется `RebootRequired` до старта новой установки, кнопка блокируется с понятным сообщением вместо непонятной ошибки API.
- **Нехватка места на диске** — проверка перед стартом скачивания.
- **Сетевые обрывы при скачивании** — ошибка по конкретному патчу не должна ронять всю партию; остальные патчи продолжают качаться/ставиться, в сводке — что упало и почему.
- **HRESULT-ошибки COM API** — маппинг известных кодов на понятные русские сообщения; для неизвестных — сырое сообщение + запись в лог (как для остальных ошибок установки в проекте).

## Тестирование

**Юнит-тесты (`Ven4Tools.Tests`)** — то, что не завязано на реальный Windows Update API напрямую (абстракция `IWindowsUpdateSource` или аналог, подставляются фейковые данные в тестах):
- Построение дерева категорий и tri-state логика чекбоксов.
- Логика EULA-гейтинга (в подтверждение попадают только патчи с непринятым EULA).
- Симметричная блокировка «идёт системная установка» в обе стороны.
- Сохранение/чтение `WindowsUpdateMode`.
- Маппинг HRESULT → сообщение.

**Не тестируется автоматически** (документированное ограничение, тот же класс, что уже принят для winget/choco/scoop): реальные вызовы `Search()`/`Download()`/`Install()` против живой Windows Update — небезопасно и бессмысленно гонять в CI (реальный патч ОС может потребовать перезагрузки раннера, займёт непредсказуемое время). UI-smoke тест (`Ven4Tools.ClientUITests`, FlaUI) ограничивается проверкой, что вкладка открывается, дерево рендерится, диалог первого входа показывается — без реального клика «Установить».

Живая проверка полного цикла — вручную, как и для остальных elevated-операций в проекте (агент не может прокликать WPF UI сам).

## Открытые вопросы для этапа реализации

- Точный список категорий `IUpdate.Categories`, которые реально приходят от Windows Update Agent, и как их лучше сгруппировать в дереве — уточнится на живом прототипе.
- UI-дизайн вкладки (макет дерева, бейджа, диалогов) — пользователь хочет увидеть визуал перед финализацией конкретных экранов; текстовый поток уже согласован.
