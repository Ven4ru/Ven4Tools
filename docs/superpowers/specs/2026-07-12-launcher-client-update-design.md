# Обновление клиента через лаунчер — ручное и тихое

## Контекст

Сейчас лаунчер (`Ven4Tools.Launcher`) умеет:
- ставить клиент с нуля и переключать версии вручную (`DownloadVersionAsync`, `MainWindow.Download.cs`);
- проверять обновление **самого лаунчера** в фоне раз в 3 часа (`UpdateBackgroundService`, `Ven4Tools.Launcher/Services/`) и предлагать его установить;
- находить актуальные версии клиента через `GitHubService` (источник истины по списку версий) и `CdnService.GetVersionInfoAsync()` (`cdn.ven4tools.ru/version.json`, подставляет более быструю ссылку на скачивание для той же версии, GitHub — фоллбек).

Чего нет: фоновая проверка **версии клиента** не запускает никакого действия — она есть только для лаунчера. Пользователь, если хочет обновить клиент, обязан вручную зайти в лаунчер и переустановить нужную версию через уже существующий, но не заточенный под «обновление» поток `DownloadVersionAsync`.

Есть отдельная, не связанная с этой задачей, подсистема — клиентский `Ven4Tools/Services/UpdateBackgroundService.cs` (другой класс с тем же именем!) — проверяет обновления winget-приложений *внутри* каталога, к обновлению самого клиента отношения не имеет.

## Цель

Добавить лаунчеру возможность обновлять клиент:
1. **Вручную** — явная кнопка проверки + существующая кнопка запуска в состоянии «обновление доступно».
2. **Тихо/автоматически** — в фоне, без участия пользователя, кроме одного диалога, если клиент запущен в момент обновления.

Режим переключается пользователем (по умолчанию — вручную, ничего не включаем без явного согласия).

## Архитектура — Вариант А (launcher-mediated, переиспользование инфраструктуры)

Обновление клиента выполняется **лаунчером**, не самим клиентом и не отдельным Windows-сервисом. Рассмотренные и отклонённые альтернативы:

- **Самообновление клиента изнутри себя** — отклонено: клиент не может атомарно заменить собственные запущенные файлы на Windows без перезапуска через внешний процесс; пришлось бы либо городить собственный мини-лаунчер внутри клиента (дублирование), либо всё равно звать существующий лаунчер — то есть тот же Вариант А, но сложнее.
- **Отдельный Windows-сервис для обновлений** — отклонено: требует новой elevated-инфраструктуры (установка/удаление сервиса, IPC), которой сейчас нет и которая не нужна — лаунчер уже персистентно живёт в трее и уже умеет делать ровно то же самое для собственного обновления.

**Вариант А** переиспользует существующий, уже проверенный код без изменений:
- `TransactionalDirectoryInstaller` — атомарная замена директории (backup/staging/swap с откатом), уже используется для установки/переключения версий клиента и для самообновления лаунчера.
- `DownloadValidator` (allow-list доверенных хостов) + `FallbackDownloader` (SHA256 fail-closed, авто-переключение на резервный источник) — уже используется в `DownloadVersionAsync`.
- `SafeZipExtractor` — безопасная распаковка во временный staging-каталог.

Механизм обновления — **полная замена каталога клиента**, не инкрементальный патч. Патчинг (delta-обновления) — перспективное направление, но осознанно отложено: сейчас нет инфраструктуры для построения diff-пакетов, а выгода (экономия трафика) не оправдывает сложность на этом этапе.

## Данные и пути

`_clientPath` (`MainWindow.xaml.cs:31-32`) — единственный источник истины о расположении клиента: по умолчанию `<папка лаунчера>\Ven4Tools_Client`, либо выбран вручную (`btnSelectFolder`), либо найден автопоиском на диске (`btnFindClient`). Путь персистится в `settings.json` (`InstallPath`) и переживает перезапуск. `UpdateBackgroundService` уже получает его как лямбду `() => _clientPath` (`MainWindow.Tray.cs:90`), то есть всегда видит актуальное значение.

Оба режима обновления (ручной и тихий) **обязаны** брать путь установки только через `_clientPath`, тем же способом, что и существующий `TransactionalDirectoryInstaller.Install(extractPath, _clientPath, token)` (`MainWindow.Download.cs:175`) — никакой отдельной логики поиска/определения пути под обновление не пишется, нестандартные (пользовательские) пути поддержаны бесплатно.

Локальные данные пользователя (`%LOCALAPPDATA%\Ven4Tools\` — профиль, логи, кэш) физически отделены от `_clientPath` и не затрагиваются полной заменой каталога клиента. Единственный теоретический риск — пользователь мог вручную указать `_installPath`, совпадающий или вложенный в папку данных. Новая защита: **перед каждым вызовом `Install()`** (и в ручном, и в тихом пути) проверяется, что `_clientPath` не равен и не вложен в `%LOCALAPPDATA%\Ven4Tools\`; при совпадении — отказ с логом, установка не выполняется.

## Поток данных

**Ручная проверка (кнопка «Проверить обновления»):**
1. Клик → перезагрузка списка версий (CDN + GitHub, как сейчас) → `DetectUpdate()` для клиента → обновление состояния `btnLaunchApp` («Обновить» вместо «Запустить», если найдена более новая версия).
2. Сам клик **не запускает скачивание** — оно стартует отдельным действием: либо пользователь жмёт `btnLaunchApp` в состоянии «Обновить», либо (в автоматическом режиме) фон запускает его сам.

**Тихий/фоновый поток (только при `AutoUpdateClient == true`):**
1. Тик таймера `UpdateBackgroundService` (существующий интервал 3ч, не меняется) → проверка версии клиента (CDN+GitHub, тот же источник, что и везде).
2. Новой версии нет → ничего не происходит.
3. Новой версии — `IsClientRunning()`:
   - **не запущен** → скачивание+установка проходят полностью без единого диалога, уведомление о результате — через существующий паттерн трей-уведомлений.
   - **запущен** → диалог с вопросом (см. «Обработка ошибок») — единственное исключение из «полностью тихо», по явному требованию не пропускать вопрос даже в автоматическом режиме.

## Панель «Настройки»

Новое модальное окно `SettingsWindow.xaml(.cs)`, открывается кнопкой `⚙ Настройки`, которая заменяет собой `Expander "Поведение лаунчера"` в главном окне. `Expander "Параметры установки"` (папка, выбор версии) остаётся на главном экране без изменений.

Содержимое окна:
- Три существующих чекбокса без изменения логики: «Проверять обновления в фоне», «Запускать свёрнутым в трей», «Запускать при старте Windows» (просто физически переезжают из главного окна).
- Новый блок «Обновление клиента» — два `RadioButton`: **«Вручную»** / **«Автоматически»**, в духе переключателя Windows Update. По умолчанию — «Вручную».

Каждый контрол сохраняет себя сразу по клику/изменению (как сейчас `ChkX_Click`) — окно не форма с Save/Cancel, а набор мгновенно применяющихся переключателей, только кнопка «Закрыть». Это сохраняет существующий в приложении паттерн, а не вводит новый.

## Обработка ошибок и краевые случаи

Переиспользуется без изменений:
- Allow-list доверенных хостов скачивания (`DownloadValidator`).
- SHA256 fail-closed с автоматическим переключением на резервный источник (`FallbackDownloader`).
- Безопасный откат при сбое установки (`TransactionalDirectoryInstaller`: backup остаётся нетронутым, если финальный swap не удался).

Новое:
- **Путь установки пересекается с папкой данных пользователя** — проверка перед `Install()`, отказ с логом (см. «Данные и пути»).
- **Сеть недоступна** (CDN и GitHub оба): при ручной проверке — лог + сообщение в UI; при тихой — только лог, следующая попытка на следующем тике таймера, без агрессивных повторов.
- **Клиент запущен в момент обновления** (единственное изменение существующей логики, `MainWindow.Download.cs:163`). Раньше — жёсткий отказ с MessageBox «закройте сами и повторите». Новое поведение, одинаковое для ручного и тихого режима:
  1. Диалог «Ven4Tools сейчас запущен. Закрыть для установки обновления?» (Да/Нет).
  2. **Да** → отправка `WM_CLOSE` в главное окно клиента (переиспользуется существующая `_isInstalling`-aware логика `OnClosing` в `Ven4Tools/MainWindow.xaml.cs` — клиент уже умеет корректно завершаться либо отказаться закрываться, если это небезопасно) → ожидание с таймаутом (~10 сек, опрос `IsClientRunning()`) → закрылся — установка продолжается; не закрылся за таймаут — отказ с логом, `Process.Kill()` **не форсируется** (обрывать чужой процесс мидтранзакции — не решение, принимаемое автоматически).
  3. **Нет** → отмена обновления, лог, ничего не трогается.

## Тестирование

- Не изменившаяся логика (`TransactionalDirectoryInstaller`, `DownloadValidator`, `FallbackDownloader`) — уже покрыта существующими тестами, новых не требует.
- Новая чистая логика (проверка пересечения пути с папкой данных, сохранение `AutoUpdateClient`) — простые unit-тесты рядом с существующими в том же тестовом проекте.
- UI лаунчера — расширение существующего `tests/Ven4Tools.UITests/LauncherSmokeTests.cs` (FlaUI): клик по `⚙ Настройки` открывает окно; переключение radio «вручную/автоматически» сохраняется и переживает перезапуск; «Проверить обновления» не виснет и обновляет состояние `btnLaunchApp`.
- Реальное скачивание + замена файлов на диске + диалог «закрыть клиента» — по риск-классификации это «рискованно» (реально трогает файловую систему и реальный процесс клиента): не автоматизируется в рутинном прогоне CI, а проходится вживую вручную один раз на тестовой сборке перед релизом фичи.

## Открытые вопросы для этапа реализации

- Точный текст и место диалога «клиент запущен» в тихом режиме (отдельное окно поверх трея vs всплывающее уведомление с кнопками) — деталь UI, не архитектуры.
- Формат уведомления об успешном тихом обновлении (баллон трея vs запись в лог лаунчера) — по аналогии с существующим уведомлением о готовом обновлении лаунчера.
