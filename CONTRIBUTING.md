# Правила разработки

## Резервная копия перед изменениями

- Перед любыми изменениями сначала проверить `git status` и создать датированный checkpoint затрагиваемых файлов в `%USERPROFILE%\_backups\`.
- В checkpoint сохранить сами файлы, `git status` и binary patch рабочего дерева.
- Если рабочее дерево уже содержит незакоммиченные изменения, считать их пользовательскими и обязательно включать пересекающиеся файлы в backup.
- Только после проверки созданной копии переходить к редактированию, форматированию, перемещению или удалению файлов.

## Пользовательский опыт

Защитные механизмы должны быть незаметны при штатной работе приложения.

- Базовый сценарий — пользователь открывает лаунчер и сразу получает рабочий интерфейс.
- Проверки безопасности не должны добавлять окна, уведомления, запросы или заметные задержки без необходимости.
- Фоновая проверка завершается тихо, если от пользователя не требуется действие.
- Сообщение допустимо только когда операция, явно запрошенная пользователем, заблокирована или без его решения продолжение небезопасно.
- Новые уведомления, автозапуск, телеметрия и фоновые сетевые действия должны быть обоснованы и отдельно проверены на влияние на запуск.
- Сбой защитного механизма не должен ломать безопасный базовый запуск; используется безопасный fallback и технический лог без чувствительных данных.

## Тесты

- Все тестовые проекты и fixtures размещаются в `tests/`.
- `tests/Ven4Tools.Tests` содержит быстрые детерминированные тесты и запускается на каждом push и pull request.
- `tests/Ven4Tools.UITests` содержит Windows UI smoke/snapshot и запускается локально либо вручную через GitHub Actions.
- Тестовый режим активируется только явно заданной переменной `VEN4TOOLS_UI_TEST=1`.
- Код test-mode не должен менять поведение обычной пользовательской сборки.

## Конфиденциальность

- В репозиторий нельзя добавлять имена пользователей, локальные абсолютные пути, содержимое рабочего стола, токены, пароли, cookie и пользовательские настройки.
- Fixtures должны содержать только публичные либо синтетические данные.
- Скриншоты перед добавлением проверяются визуально и сканируются на локальные пути и персональные данные.
- `bin/`, `obj/`, `TestResults/`, TRX, `*.actual.png`, логи и локальная проектная память не коммитятся.
